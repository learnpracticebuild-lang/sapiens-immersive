# 沉浸式深度导读引擎 · Claude Skill
> 版本: 1.0 · 封装自两本标杆书验收后的完整工作流
> 用途: 将任意原书PDF转化为高质量交互式单文件HTML阅读体验

---

## 触发条件

当用户说以下任意一种时，激活本Skill：
- "请按照阅读引擎Skill处理《XX》这本书"
- "对《XX》进行分析性阅读"
- "请处理《XX》，执行Step 0-4"
- "用沉浸式导读工作流处理《XX》"

---

## 执行前提检查

**每次开始前必读（不要跳过）：**
1. 读 `DESIGN_SPEC.md`（数据格式规范，尤其第三节的格式红线）
2. 读 `PROJECT_STATUS.md`（了解引擎当前状态和已知问题）
3. 确认原书PDF在项目目录中可读
4. 确认 `src/books/` 目录中是否已有该书的目录

## 完整执行流程

### Step 0：PDF质量预检（5分钟）

**给Claude Code的Prompt模板：**

```
请打开项目目录中《BOOK_TITLE》的PDF文件。

随机抽取以下页码，把文字原文读给我看（每处读约200字）：
- 第 [约全书1/6处] 页
- 第 [约全书2/4处] 页
- 第 [约全书5/6处] 页

报告：
1. 文字是否可直接提取（可读）？
2. 是否有大量OCR噪音、乱码或表格/图表？
3. 这本书的大致结构（几部分几章？）
4. 初步判断：这是哪种书籍类型？（见下方"书籍类型判定规则"）

不要生成任何代码，不要进行深度分析，只做质量预检。
```

**你要审核什么：**
- PDF是否可读（有些PDF是扫描图片，无法提取文字）
- 有没有严重排版问题（大量脚注干扰正文等）
- 书的结构是否清晰

**如果PDF不可读：** 停止。告知用户需要OCR版本的PDF。

### Step 1：检视阅读（30分钟）

**给Claude Code的Prompt模板：**

```
请阅读项目根目录的 DESIGN_SPEC.md，了解引擎的数据格式要求。

任务：对《BOOK_TITLE》进行检视阅读

请阅读项目目录中《BOOK_TITLE》的PDF原书，完成以下工作：

1. **全书结构**：列出所有部分/篇和每章标题，标注每章大约页数
2. **核心论点**：用3-5句话概括全书中心论点（不是"讲述了什么"，而是"作者要证明什么"）
3. **作者试图解决的问题**：这本书回答了什么核心问题？写作背景是什么年代、什么问题意识？
4. **论证策略**：作者用什么方式论证？（历史叙事/数据统计/案例分析/思想实验/操作手册/哲学论证？）
5. **关键术语清单**：列出10-20个作者赋予特定含义的术语，给出作者自己的定义（不是字典定义）
6. **全书情绪节奏**：哪些章节是震撼/论证/反思/操作性的？（这影响后续调色板设计）
7. **书籍类型初判**：根据论证策略，初步判断属于以下哪种类型（见Step 1后的类型判定规则）
8. **场景划分建议**：建议将全书划分为几个场景（8-18个）？每个场景覆盖哪些章节？

将结果输出为 BOOK_OVERVIEW.md，保存在 src/books/BOOK_SLUG/ 目录下。

重要：
- 所有内容必须基于PDF原文，不要凭记忆补充
- 引述关键段落时标注页码或章节位置
- 如果某些内容在PDF中不清晰，标注[待确认]
- 不要生成任何代码
```

**你要审核什么：**
- 核心论点是否锐利（不是"讲述了X"，而是"论证了Y"）
- 类型判定是否合理
- 场景划分是否合理（叙事型按时间/逻辑顺序，框架型按工具模块）

### Step 2：逐章分析性阅读（每Part一次对话）

#### 2A：叙事型书籍的Prompt模板

```
请阅读 src/books/BOOK_SLUG/BOOK_OVERVIEW.md 了解全书骨架。

任务：对《BOOK_TITLE》PART_N（第X-Y章）进行分析性阅读

请逐章阅读PDF原文，对每一章输出以下内容：

### 每章必须提取的7个要素：

1. **章节核心论点**（1-2句）：这一章要证明什么？
2. **完整论证链**（这是最重要的部分）：
   - 前提/假设 → 证据/案例 → 推导过程 → 结论
   - 如果一章有多条论证线，全部列出
   - 每个步骤都要包含原书的具体案例、数据或例子
3. **关键案例/故事**（完整复述，不是概括）：
   - 作者用了哪些具体历史案例、数据、思想实验？
   - 每个案例的完整叙述（保留故事性）
   - 标注在原书中的位置
4. **数据和数字**：
   - 原书引用的所有具体数据
   - 这些数据服务于哪个论证
5. **反直觉洞察**：
   - 这一章中最颠覆常识的观点是什么？
   - 作者是怎么让读者从"不可能"到"原来如此"的？
6. **章节间的逻辑衔接**：
   - 这一章怎么从上一章结论出发？
   - 这一章结论如何为下一章铺垫？
7. **可视化/交互机会**：
   - 这一章哪些内容适合做成交互组件？
   - 每个建议说明"这个交互帮助读者理解什么"

将结果输出为 CHAPTER_ANALYSIS_PARTX.md，保存在 src/books/BOOK_SLUG/ 目录下。

重要：
- 论证链必须完整，不能跳步
- 案例要保留叙事性，不要压缩成一句话摘要
- 如果论证你认为证据不足或逻辑跳跃，标注出来
- 不要生成任何代码
```

#### 2B：分析框架型书籍的Prompt模板

```
请阅读 src/books/BOOK_SLUG/BOOK_OVERVIEW.md 了解全书骨架。

任务：对《BOOK_TITLE》PART_N（第X-Y章）进行分析性阅读

这是一本分析框架型书籍。请逐章阅读PDF原文，对每一章输出以下内容：

### 每章必须提取的8个要素：

1. **章节核心框架**（1-2句）：这一章提供了什么分析工具/判断标准？
2. **框架的完整定义**：
   - 核心概念的精确定义（作者自己的措辞）
   - 适用条件和边界
   - 与相近概念的区别
3. **判断标准/决策树**：
   - 这个框架如何在实践中应用？
   - 具体的判断步骤或量化标准
   - 如何识别符合/不符合标准的情况
4. **案例演示**（带数字的完整案例）：
   - 作者用什么具体案例演示框架的应用？
   - 保留所有具体数字（比例、倍数、时间段）
   - 结果是否符合预期？异常案例如何处理？
5. **公式/比率/量化标准**：
   - 所有可量化的规则（PE<15、PB<1.5等）
   - 每个数字的来源和适用范围
6. **常见错误/反面案例**：
   - 哪些行为违反了这个框架？
   - 作者的警告和禁忌
7. **历史验证**：
   - 这个框架在历史上如何被检验？
   - 失效的条件是什么？
8. **可视化/交互机会**：
   - 哪些判断标准可以做成交互计算器？
   - 哪些对比可以做成可切换的对比模块？

将结果输出为 CHAPTER_ANALYSIS_PARTX.md，保存在 src/books/BOOK_SLUG/ 目录下。

重要：
- 框架必须包含可操作的判断步骤，不能只是概念介绍
- 保留所有具体数字
- 区分"这本书成书年代适用的标准"和"今天仍然适用的原则"
- 不要生成任何代码
```

## 书籍类型判定规则

在Step 0/Step 1完成后，根据以下规则判定书籍类型：

### 叙事型（Narrative）

**特征：**
- 作者通过历史叙事、案例故事、思想实验来推进论证
- 论证链是连续的（每章建立在上一章基础上）
- 读者价值在于"理解一个新的世界观"
- 很难跳读某一章，因为论证有严格顺序

**代表作：** 《人类简史》《枪炮病菌与钢铁》《自私的基因》《第二次机器革命》

**data.js设计重点：**
- 场景按书的叙事顺序排列
- 每个场景有丰富的narrative段落（讲故事）
- 交互组件帮助理解抽象概念（可视化数据、模拟器）
- 知识地图呈现概念演进关系
- aiReflection聚焦于"AI时代如何重新理解这些历史规律"

### 分析框架型（Framework）

**特征：**
- 作者提供可操作的分析工具、判断标准、决策框架
- 内容模块化（每章是一个独立工具，可以单独使用）
- 读者价值在于"学会一套方法论"
- 可以跳读，每章相对独立

**代表作：** 《聪明的投资者》《证券分析》《如何阅读一本书》《原则》《思考快与慢》

**data.js设计重点：**
- 场景按工具模块组织（不一定按章节顺序）
- comparison模块是核心（两种策略/两类投资者/两种思维的对比）
- 量化工具类交互（计算器、评分系统）
- 思维导图呈现框架层级结构
- aiReflection聚焦于"框架在当代的适用性修订"

### 混合型（Hybrid）

**特征：**
- 前半部分叙事（建立理论基础），后半部分框架（提供操作方法）
- 或：理论框架（Part 1）+ 历史案例验证（Part 2）

**代表作：** 《思考快与慢》（理论+实验证据）《从0到1》（理论+案例）

**data.js设计重点：**
- 前几个场景用叙事型策略（故事化、情感化）
- 后几个场景用框架型策略（对比、工具化）
- 交互组件需要既有理解类（模拟器）也有应用类（计算器）

---

## 每种类型的差异化处理策略

| 维度 | 叙事型 | 框架型 | 混合型 |
|------|--------|--------|--------|
| narrative段落 | 6-8段，讲故事 | 4-6段，讲框架 | 按场景位置决定 |
| 必须有comparison | 可选 | 必须每场景都有 | 后半部分必须有 |
| 交互组件类型 | 可视化/模拟器 | 计算器/评分器 | 两类都要 |
| deepDives侧重 | 反直觉洞察 | 框架误用案例 | 混合 |
| 知识地图形式 | 概念演进时间轴 | 决策树/层级图 | 混合 |
| aiReflection角度 | 历史规律的当代重演 | 框架的当代修订版本 | 混合 |
| 场景组织逻辑 | 时间/论证顺序 | 工具模块顺序 | 先叙事后工具 |

## 质量验收清单

### 技术验收（来自DESIGN_SPEC.md第五节，10项必验）

- [ ] **1. 构建成功**: `node build.js` 无错误，产出 `dist/BOOK_SLUG/index.html`

- [ ] **5. 术语表**: 浮动按钮可点击，术语列表完整，keyTerms不少于8个

- [ ] **8. 数据格式**: heroQuote为纯字符串，deepDives含chain/icon/title，extendedReading为字符串数组，aiReflection.palette为字符串键名
- [ ] **9. 内容深度**: 每场景≥3段narrative，≥2个deepDives，≥50%场景有keyTakeaway

## 模型选择建议

| 步骤 | 推荐模型 | 理由 |
|------|---------|------|
| Step 0: PDF预检 | **Sonnet** | 简单读取任务，不需要创造性 |
| Step 1: 检视阅读 | **Opus** | 需要识别核心论点、进行初步分类判断，需要高质量分析 |
| Step 2: 分析性阅读 | **Opus** | 最关键步骤，论证链重建和反直觉洞察需要深度理解 |
| Step 3: 批判性综合 | **Opus** | 跨章联结和批判评价需要创造性综合能力 |
| Step 4: 格式化data.js | **Sonnet** | 结构化格式转换，Opus成果已在Step 1-3完成，这步是执行 |
| 验收修复 | **Sonnet** | 针对具体问题修复，不需要全局理解 |

**总结：Step 1-3用Opus（每步独立对话），Step 4用Sonnet。**

## 注意事项

1. **一步一审核**：每步都必须你审核通过后再进下一步。不要跳步。
2. **独立对话**：每一步是一次独立的Code对话，避免上下文过长导致质量下降。
3. **先看方案再写代码**：Step 4开始前，可以先让AI输出data.js的场景划分方案（只有场景标题和覆盖章节），审核通过后再写完整data.js。
4. **不满意时用具体模板反馈**（不要说"不够好"）：
   > 场景X的问题：
   > - 具体哪里不对：[描述]
   > - 我期望看到什么：[描述]
