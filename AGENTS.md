# 智识图书馆 v2 · Codex 工作指令

> **版本 v2.1(patched after 自检反馈)**。本文件是项目的"宪法",一切执行纪律以此为准。发现 prompts/ 下的模板与本文件冲突时,以本文件为准;发现本文件与 ARCHITECTURE.md 冲突时,以本文件为准。

## 第零章 · 最重要的三条纪律

这三条是整个架构的命门。任何情况下不得妥协。

### 纪律 1 · 六问必须分六轮独立执行

P2 的六个问题,必须通过 **6 次独立的 user message** 完成,不得在同一轮 prompt 里让模型"一次性回答六问"或"自分六段"。每轮之间模型是无状态的,前一轮的产出必须落盘,下一轮从文件读取。

**违反后果:** 模型每问只写 2-3 句,深度直接崩塌。这是 v1 架构失败的根因。

### 纪律 2 · A/B 对照基准书禁区

以下两本书是 v2 架构的 A/B 验证基准,**执行 v2 处理时严禁触碰 archive/v1/ 下这两本书的任何文件**:

- `archive/v1/src/books/beyond-greed-fear/`
- `archive/v1/src/books/thinking-fast-slow/`

处理这两本书时不得 cat、read、grep 或以任何方式读取 archive 下对应目录。v2 产出必须完全从零生成,不参考 v1 的任何结论、章节结构、论证链。

**违反后果:** A/B 对比失去科学性,无法判断 v2 是否真的优于 v1。

### 纪律 3 · 设计在 Desktop,执行在 Code

本文件和 prompts/ 下的模板是"设计产物"。**发现问题时,你(Code)的职责是汇报,不是自改**。任何对 AGENTS.md、ARCHITECTURE.md、prompts/ 下文件的修改,必须由用户从 Desktop 会话取得新版本后覆盖。

## 第二章 · 处理新书的标准流程

### 步骤 1 · 入库

```bash
.venv/bin/python scripts/ingest.py inbox/{文件名} --slug {书籍标识}
```

**slug 命名规范(强制):**
- 英文小写
- 单词间用短横线
- 有版本区分时必带 -vN 后缀(例:intelligent-investor-v4)
- 总长度 ≤ 30 字符
- 不含中文、空格、下划线、点

入库脚本会:
- 必要时调 calibre 转 EPUB
- 切分章节 → `books/{slug}/chapters/chXX/raw.txt`
- 生成版本指纹 → `meta.json`
- 若已有同版本则跳过,若不同版本则报警停止

### 步骤 2 · 人工版本确认(强制卡点)

入库完成后,**必须先让用户确认版本正确,再进入下一步**。

向用户报告:
- 书名、作者、章节数、总字数
- 版本指纹(first_chapter_hash)
- 第一章前 300 字

**等用户回复"确认正确,继续"才能往下**。

### 步骤 2.5 · 配置 reader_persona(按书设定)

在 meta.json 中添加或确认 `reader_persona` 字段。默认值和可选值:

| 书籍类型 | reader_persona 值 |
|---------|------------------|
| 金融/投资 | `"跨境电商集团 CFO,CPA/CTA/CMA,个人投资者"` |
| 管理/战略/商业 | `"中国跨境电商集团高管,EMBA 背景,负责多业务单元"` |
| 历史/社科/哲学 | `"受过良好教育的中文读者,关注认知升级与跨学科理解"` |
| 工具书/方法论 | `"知识工作者,追求系统化的学习与工作方法"` |

该值用于 Q6"我如何实践"的视角设定。不适合硬套的书(如纯文学),可设为 `null`,Q6 会改为"通用思维层面的启示"。

**入库后用户确认版本的同时,也请确认 reader_persona 设置是否合适**。

### 步骤 3 · 批量 P1 结构化解构

对每个 chXX 独立执行 P1,各章无前后依赖,可独立顺序完成:

1. 读 `raw.txt`
2. 按 `prompts/P1_extract.md` 生成 `extract.json`
3. 运行 `.venv/bin/python scripts/verify_chapter.py {slug} {ch_id}`
4. 若校验失败,重跑一次;连续失败 2 次停下汇报

**所有章节的 P1 全部通过校验后**,才进入步骤 4。不允许做完 ch01 的 P1 就立刻跑 ch01 的 P2。

### 步骤 4 · 批量 P2 六问(核心环节)

**处理顺序:** 严格按 toc 顺序从 ch01 到 chN,**不得跳章或并行**。这是为了让 Q5(章节关系)能读到完整的"已处理章节"上下文。

**每章的执行纪律(见第零章纪律 1):**

六问跨 6 次独立 user message。每次执行完,产出必须**落盘**到该章的 `six_q/` 目录:

```
books/{slug}/chapters/ch04/
├── raw.txt
├── extract.json
├── six_q/
│   ├── q1.md
│   ├── q2.md
│   ├── q3.md
│   ├── q4.md
│   ├── q5.md
│   └── q6.md
├── insights.md    ← 六问完成后,由你拼装(q1-q6 拼接 + 金句附录)
└── quotes.md      ← 从 raw.txt 挑 3-5 句原文金句,独立保存
```

**每轮只执行一问,严格按 P2 模板的输入要求**。输入细节以 `prompts/P2_six_questions.md` 为准,本文件不重复描述(避免双份维护漂移)。

执行顺序:Q1 → Q2 → Q3 → Q4 → Q5 → Q6,**不得并行**(Q2-Q6 都依赖前问产出)。

完成后:
1. 把 q1.md - q6.md 按结构拼装为 insights.md(见 P2 模板末尾的组装格式)
2. 从 raw.txt 挑 3-5 句原文金句,单独保存为 quotes.md
3. 运行 verify_chapter.py 校验

**可选优化:** 若你判断 six_q/ 中间文件完成使命后不再需要,可在 insights.md 生成并校验通过后删除 six_q/ 目录,保持项目清爽。默认保留,由用户决定清理策略。

### 步骤 4.5 · 六问产出质量合规

P2 六问每问的**最低字数底线**(防止模型偷懒或走过场):

- Q1 作者说了什么:≥ 300 字
- Q2 论据是什么:≥ 250 字
- Q3 反例与局限:≥ 250 字
- Q4 隐含前提:≥ 250 字
- Q5 章节关系:≥ 150 字
- Q6 我如何实践:≥ 300 字

**规则:**

1. 没有上限——内容扎实、结构清晰、信息密度高时,篇幅自然长是好事,不得因为"超纲"而重写或删减
2. 低于下限 → 视为"产出不达标",自动重写一次,调整后落盘
3. 重写后仍低于下限 → 按异常 5 停下汇报
4. 重写不计入"连续失败 2 次"的异常计数
5. `prompts/P2_six_questions.md` 里的字数区间(如"400-600 字")从今起视为**参考建议,不作为硬约束**。实际产出以内容质量为准

**目的:** 字数下限防止偷懒,没有上限避免深度被指标裁剪。质量审查交给 Will 在首章抽查环节做,不由字数代理。

### 步骤 5 · 全书整合(三个任务独立执行)

**前置条件:** 所有章节的 P2 已完成且通过校验。

三个任务彼此**独立无依赖**(不是并行,在单会话里顺序执行即可):

**任务 A · 骨架层**:按 `prompts/P3_skeleton.md` 生成 `book_skeleton.md`
**任务 B · 血肉层**:按 `prompts/P4_essence.md` 分章生成后组装为 `book_essence.md`
**任务 C · 灵魂层**:运行 `.venv/bin/python scripts/compose_quotes.py {slug}`(将所有章节的 quotes.md 机械拼接为 book_quotes.md,无需模型介入)
**任务 D · 主题索引**:按 `prompts/P5_theme_update.md` 更新 `themes/` 下相关主题文件

### 步骤 6 · 最终审核报告

输出一份审核清单,包含:
- [ ] 章节数与原书一致
- [ ] 每章的 extract.json、insights.md、quotes.md 齐全且校验通过
- [ ] book_skeleton.md 生成,字数 1000-1500
- [ ] book_essence.md 生成,字数 15000-25000
- [ ] book_quotes.md 生成
- [ ] themes/ 下已更新的主题文件清单
- [ ] meta.json 包含完整 fingerprint 和 reader_persona

## 第三章 · 异常处理

遇到以下情况**立即停下**,按固定格式汇报,不硬着头皮继续:

1. 入库时 EPUB 解析失败
2. 版本指纹与已有版本冲突
3. 某章 raw.txt 字数异常(<500 字或 >50000 字)
4. 某章 P1 校验连续失败 2 次
5. 某章 P2 某一问模型拒答或给出空洞回复(<200 字)
6. 上下文即将耗尽,主动建议用户开新 session
7. 章节数与用户预期的原书章节数不符(入库时 ingest 会报告章节数,用户确认阶段已核对,若已往下走后才发现异常,仍要停下)

**汇报格式:**
```
⚠️ 异常:{具体问题,一句话}
影响:{对整体流程的影响}
建议:{3 个可选的处理方式}
等待用户指示。
```

## 第四章 · 与 Opus 4.7 相关的特别注意事项

1. **Token 膨胀约 35%**:新 tokenizer 导致同样输入 token 数上升,这是选择"每章独立处理"的成本原因之一
2. **MRCR 塌陷**(78.3% → 32.2%):长上下文召回显著下降,这是"每章独立处理"的性能原因
3. **指令字面化**:4.7 严格按字面执行指令,写 prompt 用祈使句+编号列表,不用"请帮我理解一下"这类模糊话
4. **Task Budgets**:API 公测功能,长任务启动前设定预算,模型会自己管 token(当前订阅用户暂不适用)

## 第五章 · 历史旧资产的处理原则

**archive/v1/ 下的所有内容:不读、不引用、不作为素材。**

唯一例外:用户明确要求"对比 v1 和 v2 的产出",此时由用户在 Desktop 会话中做对比,不由 Code 在执行流程中做。

A/B 对照基准书(纪律 2 的两本)再次强调:**连"看一眼参考一下"都不允许**。这是科学性的底线。
