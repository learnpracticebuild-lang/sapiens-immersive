## 设计原则（贯穿所有阶段）

### 原则1：格式优先级——最省token的选择
- epub > 文字版PDF > 有OCR层的扫描版PDF > 纯图片扫描版
- 纯图片扫描版：不处理。告诉用户去找文字版
- 每次Step 0必须输出格式判定和token消耗预估
- 批量处理时，先按格式分组，优先处理epub和文字版PDF

### 原则2：类型覆盖——不求穷举，用后迭代
当前已验证的类型：
- 叙事型（人类简史）
- 分析框架型（聪明的投资者）

遇到新类型时不要硬套，而是：
1. Step 1判定类型时如果不属于已有类型，标注"新类型"
2. 从最接近的已有类型出发，Step 2完成后回顾"哪些提取要素不适用"
3. 完成全流程后，将新类型的差异点补充到SKILL.md的类型判定规则中

Adler在《如何阅读一本书》中的6类分类供参考：
- 实用型（教你怎么做）→ 接近"分析框架型"
- 想象文学（小说、诗歌）→ 需要新模板（情感弧线、人物关系、主题解读）
- 故事/历史/传记 → 接近"叙事型"
- 科学/数学 → 可能需要公式渲染和证明链
- 哲学 → 需要论证结构图和概念辨析
- 社会科学 → 接近"叙事型"+"分析框架型"的混合

不需要现在就为每种类型写模板。做到第5本书时，自然会覆盖3-4种类型，届时再统一整理。

### 原则3：主动性和自进化
每完成一本书，AI应该主动做三件事（不等用户问）：

**3a. 跨书联结报告**
检查已完成的所有书的SYNTHESIS.md，找出与新书的概念交叉点。
输出格式：
```
新书《XX》与已有知识库的联结：
- 与《人类简史》的联结：[具体概念对比]
- 与《聪明的投资者》的联结：[具体概念对比]
- 建议在data.js的aiReflection中加入的跨书洞察：[具体内容]
```

**3b. 知识体系缺口建议**
基于已完成的书单，指出知识体系的空白区域，推荐下一本应该读什么。
输出格式：
```
当前知识库覆盖：[已有领域]
明显缺口：[缺失领域]
推荐下一本：《XX》，理由：[填补什么缺口]
```

**3c. Skill自检报告**
记录本次执行中遇到的问题、低效环节、可改进点。
输出格式：
```
本次执行发现的问题：
- [问题描述] → 建议改进：[具体方案]
Skill版本建议更新：[具体修改]
```

### 原则4：递减成本、递增能力
每本书应该比上一本花费更少的token和时间，同时产出质量更高。实现路径：

**成本递减机制：**
- 快速模式：Step 1-3一次性执行，不逐步审核
- 模板复用：同类型书籍的PALETTES、meta结构、custom.js组件可以跨书复用
- 批量构建：多本书共享一次build.js执行
- 跨书引用：新书的SYNTHESIS.md可以直接引用已有书的分析结论，不需要重新推导

**能力递增机制：**
- 概念索引：每本书完成后，将keyTerms汇入一个全局术语库（`skill/knowledge-index.md`）
- 主题网络：标注概念之间的关系（支持/反对/补充/应用），形成知识网络
- 阅读推荐：基于知识网络的缺口自动推荐下一本书
- 质量基线：每本新书的data.js质量不低于已有最好的那本

## 批量处理策略

当用户说"批量处理这5本书"时：

1. **Step 0批量预检**：一次性检查所有PDF/epub的格式和质量
2. **按格式分组**：epub先做，文字PDF次之，图片PDF最后（或跳过）
3. **按类型分组**：同类型的书连续处理，复用模板和组件
4. **并行准备**：一本在做Step 4时，下一本可以开始Step 1
5. **统一部署**：全部完成后一次性 `node build.js` + 推送GitHub Pages
